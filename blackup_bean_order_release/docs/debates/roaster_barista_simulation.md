🔴 **로스터**: "자, Antigravity 설치는 했어. 이제 뭐 해야 되는 거야? '주문 관리 시작할게'라고 하면 된다고?"

🟢 **PL**: "맞습니다. 하지만 지금 에이전트한테 말씀하시면 settings.yaml이 이미 존재하기 때문에, **온보딩 마법사가 자동으로 뜨지는 않아요**. U5(온보딩 마법사)가 아직 미구현이라..."

🔴 **로스터**: "그러면 처음에 Google Sheets를 내가 10개 다 만들어야 해? 그거 누가 해줘?"

🟢 **PL**: "네, 현재는 로스터 본인이 직접 만들어야 합니다. **📌 보강 포인트 #1**: Google Sheets 템플릿(13개 컬럼 헤더 + 데이터 유효성 검사)을 미리 만들어서 복사하는 방식이 필요합니다."

🟢 **PL**: "**📌 보강 포인트 #2**: 원두 코드표를 **시트 안에 별도 탭**으로 넣거나, 드롭다운으로 제공해야 합니다. 카카오톡으로 코드 보내는 건 관리 추적 불가."

## Round 3: 날짜 형식 문제

🔵 **YS(양산)**: "주문일에 '2/15'라고 쓰면 안 되나요? 매번 '2026-02-15' 이렇게 치기 귀찮은데..."

🟢 **PL**: "현재 설계에서 '2/15'를 쓰면 **⚠️ 날짜 형식 경고**가 뜹니다. 에이전트가 의미는 추론할 수 있지만 형식 경고를 표시해요."

🔵 **UOD(울산옥동)**: "저는 아예 오늘 날짜를 자동으로 넣어주면 좋겠어요. 매번 날짜 치는 것도 일이에요."

🟢 **PL**: "**📌 보강 포인트 #3**: Google Sheets의 `=TODAY()` 함수나 데이터 유효성 검사로 날짜 형식을 강제하면, 바리스타의 입력 오류를 70% 줄일 수 있습니다."

## Round 4: 대량 주문 현실

🔵 **LGR(롯데광복)**: "백화점이라 행사 때 한 번에 100kg 주문할 수도 있어요. 그런데 50kg 넘으면 경고가 뜬다고요?"

🔴 **로스터**: "롯데는 특수 매장이니까... 수량 제한을 매장별로 다르게 할 수 있어?"

🟢 **PL**: "현재 `max_order_kg`은 **글로벌 설정**(전체 매장 동일)입니다. **📌 보강 포인트 #4**: 매장별 max_order_kg 오버라이드 기능이 필요합니다. 예: `stores[].max_order_kg` 추가."

🔵 **LGR(롯데광복)**: "그리고 행사 때는 주문건수도 20~30건씩 되는데, 시트에 그만큼 치는 것도 힘들어요."

## Round 5: 배송 구역과 물류

🔴 **로스터**: "리포트에 배송 스케줄이 나오는데... 거제는 부산에서 멀잖아. 거제수월이랑 고현은 같은 날 배송하고 싶은데 이게 자동으로 되나?"

🟢 **PL**: "현재 배송 스케줄은 **배송희망일 기준 정렬**만 합니다. 구역별 묶음 배송 최적화는 v2 로드맵에 있지만 구현되지 않았습니다."

🔵 **GH(고현)**: "거제 두 매장은 항상 같은 날 배송 받고 싶어요. 그런데 수월점이랑 저희가 다른 날 주문하면 각각 나올 텐데..."

🟢 **PL**: "**📌 보강 포인트 #5**: 같은 zone의 매장을 **배송 그룹**으로 묶어서 리포트에 한꺼번에 표시하는 기능. 현재 `zone` 필드가 있으므로 구현 가능합니다."

## Round 6: 재고 관리 현실

🔴 **로스터**: "재고 20kg이라고 settings에 넣어놨는데, 주문 처리하고 나면 재고가 자동으로 줄어야 하는 거 아냐?"

🟢 **PL**: "아닙니다. 에이전트는 **읽기 전용** 원칙이기 때문에 주문을 처리해도 재고를 자동으로 차감하지 않습니다. '에티오피아 재고 10kg 남았어'라고 수동으로 말씀하셔야 합니다."

🔴 **로스터**: "그럼 매번 내가 재고 수량을 세서 말해야 돼? 그거 수작업이잖아."

🟢 **PL**: "**📌 보강 포인트 #6**: 리포트 생성 후 '오늘 total 출고량 기준 예상 잔여 재고'를 계산해서 보여주고, '이 수치로 재고 업데이트할까요?'라고 물어보는 **준자동 재고 차감** 기능."

---

## Round 7: 중복 주문 판별

🔵 **SM(서면)**: "아까 주문 넣고 수량 수정하려고 같은 행을 다시 썼는데, 중복으로 뜨나요?"

🟢 **PL**: "네, 동일 매장+원두+날짜가 2건 이상이면 **⚠️ 중복 주문** 경고가 뜹니다."

🔵 **SM(서면)**: "수정인데 중복이라고 나오면 혼란스러워요. 수정하려면 기존 행을 지우고 새로 써야 하는 거예요?"

🟢 **PL**: "**📌 보강 포인트 #7**: 바리스타 매뉴얼에 '수정 시에는 기존 행을 삭제하고 새로 입력' 가이드를 명확히 추가. 또는 '취소' 컬럼을 추가하여 취소된 주문을 표시하는 방법 고려."

---

## Round 8: 영업시간과 주문 마감

🔴 **로스터**: "보통 오후 5시에 주문 정리하는데, 그 이후에 바리스타가 시트에 추가 입력하면 어떻게 돼?"

🔵 **ESD(을숙도)**: "저희는 야간까지 영업하는데, 마감 후 주문은 다음날로 넘어가는 건가요?"

🟢 **PL**: "현재 설계에는 **주문 마감 시간** 개념이 없습니다. 에이전트 실행 시점에 시트에 있는 데이터가 전부입니다. 마감 후 추가 입력은 다음 날 리포트에 포함되거나 누락될 수 있습니다."

🟢 **PL**: "**📌 보강 포인트 #8**: settings.yaml에 `order_cutoff_time: '17:00'` 추가. 마감 후 입력된 행에 ℹ️ 경고 표시 (마감 후 입력). 제외가 아닌 **표시**로 처리."

## Round 9: 핸드폰으로 입력

🔵 **HD(해운대)**: "저는 매장에서 노트북 안 열고 핸드폰으로 시트 열어서 입력하는데, 13개 컬럼 다 치기 너무 불편해요."

🔵 **UHD(울산현대백화점)**: "맞아요, 모바일 Google Sheets에서 옆으로 스크롤하면서 치면 실수가 많아요."

🟢 **PL**: "**📌 보강 포인트 #9**: 필수 7개 컬럼만 별도 '간편 입력' 탭을 만들고, 선택 컬럼은 기본값으로 채우는 방식. 또는 Google Forms → Sheets 연동으로 모바일 최적화."

🔵 **HD(해운대)**: "Google Forms가 좋을 것 같아요! 드롭다운으로 원두 선택하고 수량만 치면 되니까."

## Round 10: 긴급 주문 처리 흐름

🔵 **SM(서면)**: "긴급 주문이면 시트에 Y 쓰는 것만으로 충분해요? 카카오톡으로 직접 연락하는 게 빠르지 않아요?"

🔴 **로스터**: "긴급은 어차피 전화로 먼저 알려주지. 시트는 기록용이야."

🟢 **PL**: "그렇다면 현재 설계의 긴급 처리 흐름은:
1. 바리스타가 **카카오톡/전화**로 긴급 연락
2. 시트에도 **기록**으로 입력 (Y 표시)
3. 리포트에 긴급 표시로 추적

**📌 보강 포인트 #10**: 매뉴얼에 '긴급 주문 = 전화 먼저 + 시트 기록'이라는 프로세스를 명시. 시트가 긴급 알림 채널이 아님을 분명히."

---

## Round 11: 다중 원두 주문 편의성

🔵 **YS(양산)**: "저는 매일 3종류 원두를 주문하는데, 매번 3행을 새로 쓰는 게 번거로워요. 어제 주문 복사해서 날짜만 바꿀 수 있나요?"

🟢 **PL**: "Google Sheets에서 행 복사 후 날짜를 수정하면 됩니다. 하지만 수량이나 배송희망일을 안 바꾸면 그대로 나갑니다."

🔵 **YS(양산)**: "실수로 어제 날짜 그대로 보내면?"

🟢 **PL**: "에이전트가 **오늘 날짜 기준** 리포트를 뽑는데, 과거 날짜 행이 있으면 이미 처리된 것으로 간주합니다. **📌 보강 포인트 #11**: '오늘 날짜가 아닌 주문이 N건 있습니다' ℹ️ 안내 추가."

---

## Round 12: 원두 재고 부족 시 대응

🔴 **로스터**: "리포트에 '재고 부족 경고'가 떴어. 에티오피아가 30kg인데 주문이 35kg이래. 그런데 내가 여기서 뭘 해야 돼? 경고만 보여주고 끝이야?"

🟢 **PL**: "현재는 경고 표시 + '생두 발주 또는 수량 조율' 권장 조치만 제시합니다."

🔴 **로스터**: "조율하려면 어떤 매장 주문을 줄여야 하는지를 알아야 하는데..."

🟢 **PL**: "**📌 보강 포인트 #12**: 재고 부족 시 해당 원두 주문 매장 목록 + 수량을 함께 표시. 예: '⚠️ ETH-YRG 재고 부족 (30kg < 35kg) — SM 15kg + HD 10kg + YS 10kg → 어떤 매장부터 조율할까요?'"

---

## Round 13: 리포트 공유

🔴 **로스터**: "리포트가 파일로 저장된다고 했는데, 이걸 바리스타들한테 보여주려면? 매번 캡처해서 카카오톡으로?"

🟢 **PL**: "현재 리포트는 `blackup_bean_order/reports/YYYY-MM/` 폴더에 마크다운 파일로 저장됩니다. 바리스타에게 공유하려면 수동 전달이 필요합니다."

🔵 **GH(고현)**: "우리는 리포트를 못 봐도 되는데, 제 주문이 제대로 들어갔는지만 알고 싶어요."

🟢 **PL**: "**📌 보강 포인트 #13**: 매장별로 '접수 확인' 기능. 예: 각 매장 Sheets에 '처리완료' 열을 추가하거나, 처리 후 'OO점 주문 N건 접수 완료' 메시지를 생성하여 공유할 수 있도록."

## Round 14: 월말 정산과 원가

🔴 **로스터**: "원가 리포트를 켜면 kg당 원가가 나온다고 했는데, 로스팅 후 수율이 반영 안 되잖아. 생두 5kg 주문이면 실제 원두는 4.2kg 정도인데."

🟢 **PL**: "맞습니다. 현재 원가 계산은 **생두 기준 단순 곱셈**입니다."

🔴 **로스터**: "바리스타 입장에서는 원가 몰라도 되는데, 내가 월말에 정산할 때 수율까지 반영된 원가가 필요해."

🟢 **PL**: "**📌 보강 포인트 #14**: 로스팅 프로파일별 수율 계수(`roasting_profiles[].yield_rate`) 추가. 라이트 0.87, 미디엄 0.85, 다크 0.82 등. 원가 계산 시 수율 반영."

---

## Round 15: 시트 동시 편집 충돌

🔵 **SM(서면)**: "서면본점은 바리스타가 3명인데요, 2명이 동시에 시트에 쓰면 충돌 안 나요?"

🟢 **PL**: "Google Sheets는 실시간 동시 편집을 지원합니다. 같은 셀을 동시에 수정하지 않는 한 문제 없습니다. 다만 같은 행을 동시에 수정하면 마지막 저장이 우선됩니다."

🔵 **SM(서면)**: "그러면 A 바리스타가 쓴 주문을 B 바리스타가 모르고 덮어쓸 수도 있잖아요?"

🟢 **PL**: "**📌 보강 포인트 #15**: 시트에 '입력자' 컬럼(K: 주문자)이 있으므로, 매장에서 '본인 이름을 반드시 입력' 규칙을 세우면 누가 쓴 건지 추적 가능. 교육 시 강조 필요."

---

## Round 16: 이전 주문 히스토리가 쌓이는 문제

🔴 **로스터**: "한 달 지나면 시트에 주문이 몇백 줄 쌓일 텐데, 에이전트가 매번 전부 다 읽어?"

🟢 **PL**: "네, 현재 에이전트는 **시트 전체**를 읽고 요청한 날짜만 필터링합니다. 데이터가 많아지면 처리 시간이 늘어날 수 있습니다."

🔴 **로스터**: "3개월 지나면 1,000줄 넘을 텐데..."

🟢 **PL**: "**📌 보강 포인트 #16**: 월별 시트 분리 또는 '아카이브' 전략 필요.
- 옵션 A: 매월 새 탭(`202602`, `202603`)으로 분리
- 옵션 B: 처리 완료된 행을 '아카이브' 탭으로 이동
- 매뉴얼에 '월초에 시트 정리' 가이드 추가"

## Round 17: 매장 추가/폐쇄

🔴 **로스터**: "내년에 신규 매장을 열거나, 기존 매장을 닫을 때는 어떻게 해?"

🟢 **PL**: "대화로 '새 매장 OO 추가해줘', '거제수월 비활성화해줘'라고 하시면 됩니다."

🔴 **로스터**: "비활성화하면 기존 데이터는? 시트에 남아있는 것도 무시되는 거야?"

🟢 **PL**: "**📌 보강 포인트 #17**: 매장 상태 필드 추가 (`stores[].active: true/false`). 비활성 매장은 리포트에서 제외하되, 설정에는 남겨서 과거 데이터 추적 가능."

## 💡 핵심 인사이트

1. **바리스타 입력 편의성이 시스템 전체 품질을 결정한다** — 에이전트의 검증이 아무리 좋아도, 입력 데이터가 나쁘면 경고만 쏟아진다.

2. **Google Sheets 템플릿 + 데이터 유효성 검사 = 가장 비용 대비 효과 좋은 개선** — 코드 한 줄 안 바꾸고 바리스타 오류를 70% 줄일 수 있다.

3. **재고 관리의 빈 구간** — 주문 처리 후 재고 자동 차감이 없어서 로스터가 수동으로 관리해야 하는 점이 가장 큰 불편.

4. **모바일 입력은 필수** — 9개 매장 중 절반 이상이 핸드폰으로 입력. 13개 컬럼 가로 스크롤은 UX 재앙.

5. **데이터 증가 대응 전략 필요** — 월 1,000줄 이상 쌓이면 처리 속도 저하. 월별 아카이브 또는 탭 분리.
