> 기존 `design_review.md`의 5회 비판을 토대로, 새로운 관점과 더 깊은 논의를 진행합니다.

## Round 1: 로스팅 프로파일 데이터 연동

### 🔴 비판 (R1 로스터 준혁 — 품질 집착파)
> "주문 시트에 로스팅단계가 '라이트/미디엄/다크' 3단계밖에 없어.
> 나는 같은 미디엄이라도 **1차 크랙 후 30초**와 **1차 크랙 후 90초**를 구분해.
> 매장이 원하는 정확한 로스팅 포인트를 모르면 매번 전화해서 확인해야 해.
> 시트에 **세부 로스팅 프로파일 코드**를 넣거나, 최소한 비고에 가이드를 줘야 해."

**제안**: 로스팅단계 컬럼을 세분화하거나, 프로파일 코드 체계 도입

### 🟡 반론 (S2 홍대점 하은 — 트렌디 매장)
> "매장 입장에서 로스팅 프로파일 코드? 우리 파트타이머가 그걸 알아?
> '미디엄' '다크' 정도가 딱 적당해. 세부 사항은 로스터가 전문가니까
> 알아서 해주는 거 아닌가? **입력이 복잡해지면 주문 자체를 안 하게 돼.**"

### 🟢 중재 (R2 로스터 미영 — 운영 효율파)
> "둘 다 맞는 말이야. 해결책은 **settings.yaml에 기본 프로파일을 매핑**하는 거야.
> `미디엄 → M-STD(기본)`, `미디엄(깊게) → M-DEEP` 이런 식으로.
> 매장은 그냥 '미디엄' 쓰면 기본값 적용, 특별 요청은 비고에 메모.
> 로스터는 내부적으로 프로파일 코드로 관리하면 돼."

### ✅ 결정: **부분 수용**

- 매장 입력 컬럼은 현행 유지 (`라이트/미디엄/다크`)
- `settings.yaml`에 로스팅 프로파일 매핑 테이블 추가:
  ```yaml
  roasting_profiles:
    라이트: "L-STD"
    미디엄: "M-STD"
    다크: "D-STD"
  ```
- 비고 컬럼에 세부 요청 기재 시 에이전트가 리포트에서 **별도 표기**
- 컬럼 추가 없이 기존 구조 유지 → 매장 부담 최소화

## Round 2: 배치 합산 로스팅 최적화

### 🔴 비판 (R2 로스터 미영 — 운영 효율파)
> "강남점이 에티오피아 5kg, 홍대점도 에티오피아 3kg 주문했으면
> 나는 한꺼번에 **8kg 배치로 로스팅**하고 싶어.
> 그런데 리포트는 매장별로만 보여주니까, 같은 원두+같은 로스팅단계를
> **합산한 '생산 배치 뷰'**가 없으면 내가 수작업으로 계산해야 해."

**제안**: 리포트에 "생산 배치 합산" 섹션 추가 (원두+로스팅단계별 합산)

### 🟡 반론 (S1 강남점 지수 — 대형 매장)
> "생산 효율화는 로스터 쪽 이슈인데, 그걸 위해 리포트가 복잡해지면
> 나도 읽기 어려워. 매장 입장에서는 '내 주문 상태'만 보이면 돼.
> **생산 배치 뷰는 관리자용 별도 섹션**으로 분리해줘."

### 🟢 중재 (R4 로스터 소연 — 고객 소통파)
> "맞아, 리포트를 audience별로 나누는 게 좋겠어.
> 현재 리포트 3종(일일 요약, 원두별 합산, 배송 스케줄) 중
> '원두별 총 주문량' 테이블에 로스팅단계별 소계 행을 추가하면
> 별도 섹션 없이도 배치 판단 가능해."

### ✅ 결정: **수용**

- "원두별 총 주문량" 테이블에 **로스팅단계** 컬럼 추가
- 같은 원두라도 로스팅단계가 다르면 별도 행으로 표시:
  ```markdown
  | 원두명           | 코드     | 로스팅  | 총 수량(kg) | 재고(kg) | 과부족 |
  |-----------------|----------|--------|------------|---------|--------|
  | 에티오피아 예가체프 | ETH-YRG | 미디엄  | 8          | 30      | +22    |
  | 에티오피아 예가체프 | ETH-YRG | 라이트  | 3          | (합산)   | -      |
  | 콜롬비아 수프리모  | COL-SUP | 다크    | 5          | 15      | +10    |
  ```
- 로스터는 이 테이블로 배치 계획 수립 가능

## Round 3: 소량 다품종 주문 처리

### 🔴 비판 (S2 홍대점 하은 — 트렌디 매장)
> "우리 매장은 한 달에 한두 번 **새 원두를 소량(1~2kg) 테스트 주문**해.
> 근데 `settings.yaml`의 beans 목록에 없는 원두는 어떻게 되는 거야?
> '과테말라 안티구아' 2kg 주문하고 싶은데, 목록에 없으면 에이전트가 무시해?
> **신규 원두 주문을 처리하는 방법**이 필요해."

**제안**: 미등록 원두 주문 허용 + 경고 플래그

### 🟡 반론 (R3 로스터 동현 — 리스크 관리파)
> "아무 원두나 주문하게 하면 난리나. 우리가 취급하지도 않는 원두를
> 매장이 멋대로 넣으면 어떡해? 목록에 없는 건 **에러**로 처리하는 게 맞아.
> 신규 원두는 관리자가 사전에 등록해야 해."

### 🟢 중재 (R5 로스터 태준 — 데이터 분석파)
> "둘 다 일리가 있는데, 절충안이 있어. 미등록 원두는 **주문 자체는 접수**하되,
> 리포트에서 🟡 **'미등록 원두'** 경고로 별도 표시. 관리자가 확인 후
> beans 목록에 추가할지 판단. 이러면 주문은 놓치지 않고, 무분별한 추가도 방지돼."

### ✅ 결정: **수용**

- `beans` 목록에 없는 원두 코드 → **주문 접수는 허용**
- 리포트의 교차검증에 **"미등록 원두 감지"** 항목 추가:
  ```markdown
  | 항목 | 내용 | 결과 |
  |------|------|------|
  | 미등록 원두 | 홍대점: 과테말라 안티구아 (GUA-ANT) | 🟡 미등록 — 관리자 확인 필요 |
  ```
- 교차검증 통과 조건에서는 제외 (경고만 표시, 차단하지 않음)

## Round 4: 자동 알림 및 매장 피드백 채널

### 🔴 비판 (S3 성수점 민규 — IT 친화적)
> "시트에 주문 넣고 나서 **아무런 피드백이 없다는 게 불안해.**
> 카카오톡이든 Slack이든, '주문 접수 완료' 알림이 와야지.
> 현재 설계에선 매장이 수동으로 리포트를 확인해야 하는데,
> 관리자가 언제 리포트를 보낼지도 모르니까 하루종일 기다릴 수도 있어."

**제안**: 에이전트가 리포트 생성 후 자동으로 매장에 알림 전송 (카카오/Slack/이메일)

### 🟡 반론 (R4 로스터 소연 — 고객 소통파)
> "알림 자체는 무조건 좋아. 하지만 v1에서 카카오 API나 Slack webhook을
> 연동하면 인증·설정·유지보수 부담이 커. 지금 구조는 '브라우저로 Sheets 읽기'만
> 하는 단순한 구조인데, 외부 API 호출까지 넣으면 복잡도가 확 올라가지."

### 🟢 중재 (R1 로스터 준혁 — 품질 집착파)
> "그러면 **최소한의 피드백 방법**으로, 리포트를 Google Docs나
> 별도 공유 Sheets에 자동 기록하는 건 어때? 매장이 URL만 알면
> 언제든 최신 리포트를 확인 가능. 추가 API 연동 없이 가능하잖아.
> 본격 알림은 v2에서 하고."

### ✅ 결정: **부분 수용 (v1.5 계획)**

- v1: SKILL.md에 "리포트 생성 후 관리자가 직접 매장에 전달" 가이드 유지
- v1.5: 리포트를 **공유 Google Docs에 자동 기록** → 매장이 URL로 조회 가능
  - `settings.yaml`에 `report_doc_url` 추가
- v2: 카카오/Slack/이메일 알림 연동
- 교차검증에서 리포트 기록 성공 여부 확인 가능하도록 설계

## Round 5: 다국어 지원 및 원두 영문명

### 🔴 비판 (S4 이태원점 세라 — 외국인 고객 多)
> "우리 매장은 외국인 바리스타도 있어. 시트에 한글로만 원두명 쓰면
> 외국인 직원이 **뭘 주문하는지 몰라.** 최소한 원두명은
> **한글/영문 병기**가 필요해. 국제 기준 원두 코드명도 있으면 좋고."

**제안**: `beans` 목록에 영문명(`name_en`) 필드 추가, 시트 입력 시 영문명도 허용

### 🟡 반론 (S5 연남점 우진 — 소규모 매장)
> "컬럼 늘리는 건 반대야. 우리 같은 소규모 매장은 시트 입력 자체가 부담인데
> 영문명까지 넣어야 하면 더 귀찮아져. 다국어 지원은
> 이태원점 같은 특수 케이스 아닌가?"

### 🟢 중재 (R5 로스터 태준 — 데이터 분석파)
> "시트 입력에는 영향 없어. `settings.yaml`의 beans 목록에
> `name_en` 필드만 추가하면, 에이전트가 리포트 생성할 때
> 자동으로 영문명을 병기할 수 있어. **입력은 변화 없고 출력만 풍부해지는 거야.**"

### ✅ 결정: **수용**

- `settings.yaml` beans에 `name_en` 필드 추가:
  ```yaml
  beans:
    - { name: "에티오피아 예가체프", name_en: "Ethiopia Yirgacheffe", code: "ETH-YRG", stock_kg: 30 }
    - { name: "콜롬비아 수프리모", name_en: "Colombia Supremo", code: "COL-SUP", stock_kg: 15 }
  ```
- 매장 시트 입력은 변경 없음 (한글 or 코드)
- 리포트에서 원두명 한/영 병기: `에티오피아 예가체프 (Ethiopia Yirgacheffe)`

## Round 6: 간소화된 주문 방식 (소규모 매장 배려)

### 🔴 비판 (S5 연남점 우진 — 소규모 매장)
> "13개 컬럼이라고? 우리 매장은 원두 2종류만 쓰고 매주 같은 양 주문해.
> 그런데 매번 주문일, 매장코드, 매장명, 원두코드, 원두명... 다 채워야 해?
> **반복 주문을 간단하게 처리하는 방법**이 없으면 써먹기 어려워."

**제안**: 반복 주문 템플릿 또는 '이전 주문과 동일' 간편 입력 기능

### 🟡 반론 (R3 로스터 동현 — 리스크 관리파)
> "간편 주문? 그러면 실수로 지난주 주문이 그대로 들어올 수 있어.
> 수량 변동 없이 무조건 복사면 재고가 맞지 않는 상황이 생겨.
> 주문은 매번 명시적으로 입력하는 게 안전해."

### 🟢 중재 (S3 성수점 민규 — IT 친화적)
> "Google Sheets에서 해결 가능해. 시트에 **'정기 주문' 탭**을 만들고
> 그 탭에 기본값을 세팅해두면, 직원이 매주 그 탭을 '이번 주 주문' 탭으로
> 복사만 하면 돼. 시스템 변경 없이 시트 운영 방식으로 해결이야.
> SKILL.md에 가이드만 추가하면 충분해."

### ✅ 결정: **부분 수용 (운영 가이드)**

- 시스템 기능 추가 없음
- **SKILL.md에 운영 가이드 추가**: "정기 주문 매장은 '기본 주문' 탭 활용 권장"
- `examples/sample_sheet_data.md`에 **정기 주문 탭 예시** 포함
- 필수 7개 컬럼만 입력 필수, 선택 6개는 생략 가능 (이미 설계에 반영됨)

## Round 7: 생두 수급 리스크 및 가격 변동 관리

### 🔴 비판 (R3 로스터 동현 — 리스크 관리파)
> "에티오피아 생두 가격이 2주 만에 30% 올랐어! 주문 들어온 대로 생산하다가
> 적자 나면 어떡해? 현재 리포트에는 **원가 정보가 전혀 없어.**
> 최소한 원두별 **대략적 단가**를 settings에 넣고,
> 리포트에서 '이번 주문 총 예상 원가'를 보여줘야 손익 파악이 돼."

**제안**: `settings.yaml`에 원두별 원가(cost_per_kg) 추가, 리포트에 예상 원가 표시

### 🟡 반론 (S1 강남점 지수 — 대형 매장)
> "원가 정보는 로스터 내부 관리 사항이지, 매장에 노출되면 안 되는 거 아닌가?
> 리포트를 매장과 공유할 수도 있는데 원가가 찍혀 있으면 곤란해."

### 🟢 중재 (R2 로스터 미영 — 운영 효율파)
> "걱정할 필요 없어. 리포트는 '관리자용'이야. 매장에 보내는 건
> 매장별 주문 내역만 발췌해서 보내면 돼. 관리자 리포트에 원가 정보가
> 있는 건 당연하고, 오히려 없으면 경영 판단이 안 돼.
> 단, `settings.yaml`에서 **선택적으로 활성화**하면 유연해."

### ✅ 결정: **수용 (선택 기능)**

- `settings.yaml`에 원가 관련 설정 추가:
  ```yaml
  enable_cost_report: true  # 원가 리포트 활성화 여부

  beans:
    - { name: "에티오피아 예가체프", code: "ETH-YRG", stock_kg: 30, cost_per_kg: 25000 }
  ```
- `enable_cost_report: true`일 때만 리포트에 "예상 원가" 섹션 출력
- 매장 공유용 리포트에는 원가 섹션 미포함 (관리자 전용)

## Round 8: 데이터 무결성 및 입력 검증 강화

### 🔴 비판 (R5 로스터 태준 — 데이터 분석파)
> "매장 직원들의 입력 실수가 걱정돼. `수량(kg)`에 '다섯' 이라고 한글로 쓰거나,
> 배송희망일에 '내일모레'라고 쓰면? 날짜 형식이 `2026-02-15`인데
> `2/15`이나 `Feb 15`로 쓰는 매장도 있을 거야.
> **입력 값 유효성 검증**이 교차검증에 포함되어야 해."

**제안**: 교차검증에 데이터 타입/형식 검증 추가

### 🟡 반론 (S5 연남점 우진 — 소규모 매장)
> "날짜 형식 강제하면 입력하다 짜증나서 안 쓰게 될 수 있어.
> Google Sheets 자체 데이터 유효성 검사 기능을 쓰면 시트 단에서
> 입력 제한이 가능한데 굳이 에이전트까지 검증해야 하나?"

### 🟢 중재 (S3 성수점 민규 — IT 친화적)
> "시트 자체 데이터 유효성 검사는 좋은 방법이야. 하지만 모든 매장 시트에
> 동일하게 적용한다는 보장이 없잖아. **2중 안전장치**로,
> 에이전트가 읽을 때도 형식 검증을 하는 게 맞아.
> 비정상 데이터는 차단이 아니라 **경고와 함께 해당 행을 별도 표시**하면 돼."

### ✅ 결정: **수용**

- Google Sheets에 **데이터 유효성 규칙** 설정을 **운영 가이드(SKILL.md)**에 추가
- 에이전트 교차검증에 **입력 형식 검증** 항목 추가:
  ```markdown
  | 항목 | 내용 | 결과 |
  |------|------|------|
  | 날짜 형식 | 모든 날짜가 YYYY-MM-DD 형식인지 | ✅ / ⚠️ |
  | 수량 숫자 | 수량(kg)이 양의 숫자인지 | ✅ / ⚠️ |
  | 필수 컬럼 | 필수 7개 컬럼 누락 없는지 | ✅ / ⚠️ |
  | 원두코드 일치 | 원두코드-원두명 매핑이 settings와 일치하는지 | ✅ / 🟡 |
  ```
- 비정상 행은 **리포트 하단에 "검증 주의사항"**으로 별도 목록 표시

## Round 9: 배송 구역화 및 배송 루트 최적화

### 🔴 비판 (R2 로스터 미영 — 운영 효율파)
> "강남점 → 성수점 → 이태원점 순서로 배송하는 것과
> 강남점 → 이태원점 → 성수점 순서는 시간·비용이 다른데,
> 현재 배송 스케줄은 **날짜별 나열만** 해줘. 배송 동선을 고려한
> **구역별 그룹핑**이라도 있으면 배송 효율이 올라가."

**제안**: 매장에 배송 구역(`zone`) 속성 추가, 배송 스케줄을 구역별 그룹핑

### 🟡 반론 (R1 로스터 준혁 — 품질 집착파)
> "배송 루트 최적화까지 가면 scope이 너무 커져. 현실적으로 우리
> 배송 기사가 알아서 동선 짜는 건데, 시스템이 루트까지 정해주면
> 오히려 현장과 안 맞을 수도 있어."

### 🟢 중재 (S4 이태원점 세라 — 외국인 고객 多)
> "루트 최적화까지는 과한데, 매장에 **지역 태그** 정도는 있으면 좋겠어.
> 배송 스케줄에서 '강남권', '마포권' 이렇게 묶어서 보여주면
> 배송 기사도 한눈에 파악 가능하잖아. 복잡한 로직 없이 태그만."

### ✅ 결정: **부분 수용**

- `settings.yaml` stores에 `zone` 필드 추가:
  ```yaml
  stores:
    - { name: "강남점", code: "GN", sheet_url: "...", zone: "강남권" }
    - { name: "성수점", code: "SS", sheet_url: "...", zone: "성동권" }
    - { name: "이태원점", code: "IT", sheet_url: "...", zone: "용산권" }
  ```
- 배송 스케줄 테이블에 `구역` 컬럼 추가, 구역별 소계 그룹핑
- 최적 루트 계산은 **scope 밖** (v2+ 검토)

## Round 10: 시스템 확장성 및 장기 데이터 관리

### 🔴 비판 (R5 로스터 태준 — 데이터 분석파)
> "매일 주문 리포트를 생성하면 6개월 뒤에는 리포트가 180개야.
> **과거 리포트 아카이빙, 검색, 통계**를 어디서 하지?
> 지금 구조로는 리포트가 날짜별로 쌓이기만 하고 찾아볼 방법이 없어.
> 최소한 **월별 폴더 구조**나 **리포트 인덱스**가 필요해."

**제안**: 리포트 아카이빙 구조(폴더/인덱스) 정의 + 월간 요약 리포트

### 🟡 반론 (R4 로스터 소연 — 고객 소통파)
> "장기 데이터 관리는 중요하지만, 에이전트가 과거 리포트를 관리하는 건
> 현재 '주문 정리' scope에서 벗어나. Google Drive 폴더 정리는
> 관리자가 하는 게 맞고, 에이전트는 오늘 리포트만 잘 만들면 돼."

### 🟢 중재 (S3 성수점 민규 — IT 친화적)
> "에이전트가 리포트 파일명에 **날짜를 포함**하는 건 이미 할 수 있어.
> 추가로 `reports/YYYY-MM/` 폴더 구조만 정해주면 관리자가 정리하기도
> 편하고, 나중에 검색도 가능해. 인덱스 파일까지는 v2에서 하자.
> 현재는 **네이밍 규칙 + 폴더 구조 가이드**로 충분해."

### ✅ 결정: **부분 수용**

- 리포트 파일 네이밍 규칙 정의:
  ```
  daily_report_2026-02-15.md
  ```
- 권장 폴더 구조 가이드 (SKILL.md에 추가):
  ```
  reports/
  ├── 2026-02/
  │   ├── daily_report_2026-02-15.md
  │   ├── daily_report_2026-02-16.md
  │   └── ...
  └── 2026-03/
      └── ...
  ```
- 월간 요약 리포트 → **v2 로드맵 등록**
- 리포트 인덱스/검색 → **v2 로드맵 등록**

```yaml
roasting_profiles:           # Round 1
  라이트: "L-STD"
  미디엄: "M-STD"
  다크: "D-STD"

enable_cost_report: false    # Round 7 (기본 비활성)

# ===== beans 확장 =====
beans:
  - name: "에티오피아 예가체프"

    code: "ETH-YRG"
    stock_kg: 30
    cost_per_kg: 25000                  # Round 7

# ===== stores 확장 =====
stores:
  - name: "강남점"
    code: "GN"
    sheet_url: "https://docs.google.com/spreadsheets/d/SHEET_ID_GN"
    zone: "강남권"                       # Round 9

# ===== 기존 유지 =====
min_lead_days: 2
reports:
  - daily_summary
  - bean_totals
  - delivery_schedule
```
